# AIGril 设计平台初步方案

## 产品拆分

AIGril 现在应该拆成两套系统：

1. `AIGril Runtime`
   - 负责虚拟人对话、表情、动作、演出执行
   - 只消费已经标准化的角色包
   - 不承担大规模编辑任务

2. `AIGril Studio`
   - 负责用户创作自己的虚拟人
   - 负责形象、性格、声音、歌曲、动作、作品模板、发布
   - 负责把设计结果发布到 Runtime

## 设计平台的最小闭环

### 1. 角色定义

- 形象原型：脸、发型、服装、风格线、动作气质
- 性格协议：口吻、边界、主动性、情绪基调
- 声线协议：说话音色、语速、唱歌风格

### 2. 资源装配

- 用“作品”组织歌曲、伴奏、歌词、动作、表情模板
- 用“角色”组织长期人格与视觉身份
- 用“演出模板”组织可运行的输出能力

### 3. 发布接入

- 生成角色包
- 对 Runtime 兼容性做校验
- 发布、版本记录、回滚

## 初版信息架构

### 产品蓝图

先回答三个问题：

- 为什么要分成 Runtime 和 Studio
- 角色包协议长什么样
- V1 先做哪三件事

### 形象工坊

- 角色原型
- 视觉风格
- 运行时兼容度

### 性格与声音

- 人格模板
- 对话协议
- 声线模板

### 歌曲与动作

- 资源槽位
- 作品模板
- 演出模板

### 发布接入

- 包结构
- 校验器
- Runtime 发布

## 我建议的 V1 开发顺序

1. 角色包协议
2. 角色定义页
3. 作品模板与资源槽位
4. 发布校验与 Runtime 接入

先不要急着做：

- 大型协作平台
- 创作者市场
- 审核流和复杂权限系统

## 参考信号

这些不是照抄，而是提炼它们各自最强的结构能力：

- MetaHuman：高保真角色创建 + 与运行时强绑定
- Ready Player Me：低门槛创建 + 跨应用分发
- VRoid Studio：创作者视角的精细形象编辑
- Character Creator：围绕角色组织服装、动作、资产
- 腾讯云数字人：流程化生产与发布接入
- Canva / Figma：模板系统、设计资产中台、版本意识
